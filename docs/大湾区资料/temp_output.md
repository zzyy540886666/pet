## **一、作品简介**

本作品基于OpenHarmony/SwanLinkOS操作系统，以电子萌宠为核心方向，构建集AI
智能交互、全息浮空显示、双端协同、IOT
感知、永久记忆、情绪共鸣于一体的虚拟数字宠物产品。项目采用手机控制端 +

"有灵魂的数字伙伴"

### **二、问题分析**

1.  **用户与市场痛点****与产品核心解决问题**

现代用户对宠物陪伴需求持续提升，但日常照料时间有限、养护成本较高。现有虚拟宠物产品功能单一，缺乏长期记忆、情绪变化与成长逻辑，情感连接薄弱。智能硬件设备普遍存在生态封闭、数据孤立、跨设备协同能力不足等问题，难以形成连贯的使用体验。

项目需要构建具备自主情绪、持续成长、深度交互的虚拟宠物体系，实现多设备间的流畅协同，同时保证互动数据可长期留存，让虚拟宠物具备稳定人格与持续情感价值，并在无外接硬件、无网络环境下仍可稳定运行。

### **三、现有技术短板**

#### **虚拟宠物交互与人格技术短板**

**· 记忆机制缺失或浅层化**

现有虚拟宠物大多仅支持短期会话记忆，不具备长期记忆、语义记忆与用户画像能力，无法记住历史互动、主人偏好与关键事件，每次交互都像
"初次见面"，难以建立稳定情感连接。

**· 情绪与成长模型不成熟**

多数产品只有简单表情切换，没有完整的情绪生成、情绪转换、情绪影响机制；性格固定不变，无法随交互频率、互动方式、环境变化自主演化，缺少真实生命的成长感与差异性。

**· 语音交互体验薄弱**

语音识别准确率低、合成音色生硬，缺少离线语音能力；对话逻辑简单、无角色一致性，容易出现人设崩塌，无法做到自然聊天、情绪回应与主动关心。

#### 多设备协同与硬件联动短板

**· 分布式能力缺失**

不同设备之间数据不通、画面不同步，无法实现
"手机交互、大屏展示、硬件感知" 的协同体验，与鸿蒙分布式技术理念脱节。

**· 南向感知与数字宠物脱节**

传感器数据仅做简单显示，没有与宠物状态、情绪、行为深度绑定；环境温度、湿度、人体感应等信息无法转化为宠物的真实反应，智能感知流于形式。

**· 全息显示落地难度高**

现有全息虚拟形象方案普遍存在渲染不同步、视角割裂、延迟高问题，缺少轻量化、可在开源鸿蒙设备上运行的四向同步渲染引擎，工程化落地成本高。

#### 本地存储与离线可靠性短板

**· 持久化存储方案不完善**

互动记录、对话历史、成长数据多采用缓存或轻量存储，易丢失、不支持检索，无法满足
"永久存储、可追溯、可回顾" 的强需求。

**· 严重依赖云端服务**

AI
对话、语音、逻辑计算全部放在云端，断网即不可用；缺少本地降级策略、离线交互库、断点续传机制，产品稳定性与可用性极差。

#### 开源鸿蒙生态适配短板

**· 北向应用与南向设备开发分离**

多数项目要么只做 APP，要么只做硬件，缺少完整的 "北向应用 + 全息显示端 +
南向感知" 三端一体化开发范式，技术链路不闭合。

**· 星闪、分布式数据等鸿蒙特色能力使用不足**

现有作品很少深度使用近场通信、分布式数据库、设备管理等原生能力，未能体现开源鸿蒙在物联网与跨端协同上的技术优势。

### 后端服务器搭建

**· 服务器的定位与核心价值**

在智能宠物关怀系统的三层架构中，服务器作为连接OpenHarmony终端与前端应用的核心枢纽，承担着异构数据中转、数据持久化存储、双向指令转发及设备在线状态维护的关键角色。系统中同时存在三类关键数据：温湿度传感器的周期性遥测数据、宠物图片
/
视频的高带宽媒体流、以及前端下发的控制指令，三类数据对传输的可靠性、延迟性要求差异显著，直接传输难以兼顾效率与稳定性。

本服务器设计针对不同数据类型的特性，采用分层通信架构：温湿度数据基于MQTT协议实现温湿度数据的可靠上报与指令下发，保障传感器数据的完整性；图像传输传输基于
WebSocket
协议实现音视频流的低延迟实时传输，保障监控画面的流畅性；同时通过数据库实现温湿度历史数据的持久化，为宠物环境分析提供数据支撑。此外，服务器通过心跳检测机制维护终端与客户端的在线状态，实现断网重连与异常告警，保障系统的长期稳定运行，为整个系统的实时监护与远程控制功能提供可靠的后端支撑。

**· 本设计的目标与主要内容**

本服务器设计的核心目标，是构建一套轻量、高并发、低延迟的后端服务，解决
OpenHarmony
终端与前端应用之间的多协议通信、数据分流与状态管理问题。后续内容将围绕服务器的模块设计、通信流程、业务逻辑及数据存储方案展开详细阐述，为整个智能宠物关怀系统的功能实现与稳定运行提供后端保障。

**· 整体设计流程图伪代码**

该伪代码定义了开源鸿蒙智能宠物关怀系统的中转服务器核心运行逻辑，服务器作为连接智能硬件设备与用户前端应用的核心枢纽，承担数据接收、存储、转发与设备管理的全部功能。

首先，服务器预先定义了基础运行参数，包括 MQTT 通信端口、WebSocket
通信端口、数据存储数据库，以及用于记录在线智能设备和用户前端的列表。服务器启动后，会依次启动
MQTT 服务与 WebSocket
服务，完成数据库初始化，并开启心跳监控机制，随后持续监听所有设备与前端的网络连接请求。

在运行过程中，服务器通过 MQTT
协议接收智能设备上传的温湿度传感数据，将数据持久化存储到数据库中，同时把实时数据广播推送给所有在线的用户前端。针对用户前端下发的远程控制指令，服务器会解析指令目标，通过
MQTT 协议将指令精准转发至对应的开源鸿蒙智能设备。

针对图片、视频等媒体数据，服务器通过 WebSocket
协议进行处理：若数据来自智能设备，则将媒体流实时广播给所有用户前端；若数据来自用户前端，则解析为控制指令并转发至目标设备。

此外，服务器通过独立的心跳监控模块持续循环运行，定时更新智能设备与用户前端的在线状态，保障系统连接的稳定性与可靠性，实现整个智能宠物关怀系统的稳定通信与数据交互。

### 芯片设计原理

左侧 PMU 电源管理单元与 CMU
时钟管理单元，为芯片提供稳定供电与时钟基准：PMU 通过基准源、LDO
稳压器及过压欠压保护电路，实现电源域管理与异常防护；CMU
通过晶振、锁相环及时钟分频电路，为
CPU、外设和射频模块提供多域多频率时钟，保障系统在高性能与低功耗模式下稳定运行。

中间 SoC 核心以**高速总线矩阵**为数据交互枢纽：带缓存的 CPU
作为主设备发起指令与数据访问；DMA
控制器独立完成外设与存储间的数据搬运，降低 CPU
负载；安全子系统提供硬件级安全防护。内置 RAM/ROM
为系统运行提供高速数据空间，通过双路 QSPI 总线扩展 Flash 与
PSRAM，支撑固件存储与系统运行；看门狗、定时器、RTC
等基础外设通过总线接入，实现系统调度与状态监测。

无线通信子系统依托总线完成 Wi-Fi、蓝牙 5.3 与星闪 SLE1.0
的协议处理，通过 2.4G
射频前端实现信号收发，支撑多协议连接。右侧外设接口矩阵作为总线从设备，通过
CPU 或 DMA
控制，实现对按键、蜂鸣器、显示屏、传感器等外设的驱动与数据交互。

"活的伙伴" 这一设计目标。

# 六**、**测试分析

#### **测试环境**

测试覆盖纯模拟器环境、OpenHarmony 实机、SC-3568HA 开发板、Hi3863
星闪开发板等组合，采用局域网与近场通信，验证纯软件、软硬结合两种部署模式下的表现。

#### **功能与性能表现**

宠物动画、触摸互动、语音交互、数据存储、双端同步、传感器采集、情绪响应、离线运行等核心功能均稳定可用。系统动画流畅、交互延迟低、启动速度快，长时间运行无明显卡顿与异常。

**动画帧率（≥25fps，达标）**动画帧率是影响虚拟宠物互动流畅度的核心指标，≥25fps
的标准可保证宠物进食、玩耍、亲昵、对话等全场景动画渲染连贯自然，无掉帧、卡顿或画面撕裂问题，还原真实宠物的灵动表现，保障沉浸式互动体验的沉浸感与真实感。

**双端同步延迟（≤50ms，达标）**双端同步延迟直接关系到跨设备联动的体验连贯性，≤50ms
的低延迟表现，可确保手机端与硬件端的宠物状态、互动数据实时同步，用户在任一设备的操作都能在另一端即时呈现，实现无感知的跨端联动，避免状态不一致带来的体验割裂。

**应用启动时间（≤3s，达标）**应用冷启动时间直接影响用户的使用效率，≤3s
的快速启动表现，大幅降低了用户进入互动场景的等待成本，用户可快速唤起 APP
与宠物互动，提升产品的即时可用性，减少操作门槛与等待焦虑。

**本地存储占用（≤100MB，达标）**本地存储占用控制在 100MB
以内，既满足了用户互动记录、宠物成长档案、梦境数据等内容的本地存储需求，又不会对用户设备造成存储压力，同时保障了应用的轻量化安装与长期稳定运行，适配更多用户设备场景。

# **七、**作品总结

#### **作品特色与创新点**

**1、全息浮空显示：**四向 Canvas 渲染配合光学结构，实现 3D
浮空宠物效果，展示效果突出。

**2、多设备分布式协同**：手机控制、全息展示、硬件感知三端联动，体验连贯统一。

**3、双层永久记忆：**本地 RDB+AI
三层记忆，可长期留存互动内容，宠物具备持续人格。

**4、情绪与成长闭环：**宠物拥有自主情绪、性格成长、梦境生成，情感陪伴维度丰富。

**5、全链路离线降级：**无硬件、无网络、无 AI
均可稳定运行，适用场景更广。
