# 虚拟数字宠物 — 2.5D 功能需求文档

> 项目名称：**鸿蒙虚拟数字宠物（HarmoPet）**  
> 视觉策略：**由 3D 调整为 2.5D 轻量交互表现**  
> 开发策略：**手机 / Previewer 单端优先落地**，以稳定演示、轻资源、易维护为第一目标  
> 适用平台：**HarmonyOS NEXT / ArkTS 严格模式 / Stage 模型 / API 12+**

---

## 一、调整背景

### 1.1 调整目标

原方案以 3D / 全息表现为核心亮点，但在当前阶段存在以下问题：

- 3D 资源制作与接入成本较高，影响 MVP 推进速度。
- Previewer、模拟器与不同设备上的一致性较难保证。
- 真实 3D 渲染、模型加载、动画资源管理会提高维护复杂度。
- 当前项目更需要先稳定完成“可交互、可成长、可陪伴”的核心体验。

因此将宠物表现方案调整为 **2.5D 轻量视觉方案**：

- 保留“有生命感、有层次感、有互动反馈”的视觉目标。
- 不再依赖重型 3D 模型作为主链路。
- 使用 **精灵图、序列帧、多层贴图、阴影、视差、轻旋转、浮动动画** 来实现空间感。
- 保证手机端、Previewer 与后续硬件扩展都能共享同一套核心交互逻辑。

### 1.2 2.5D 定义

本项目中的 2.5D，指的是：

- 宠物仍然以 **2D 资源** 为主；
- 通过 **前中后景分层**、**缩放**、**位移**、**阴影**、**小角度旋转**、**动态光效** 模拟立体层次；
- 用户感知上接近“会呼吸、会转头、会浮动”的数字角色；
- 工程实现上仍然保持 **ArkUI 标准组件可控、可降级、可调试**。

## 二、产品目标

### 2.1 核心目标

打造一个以“陪伴感”为核心的鸿蒙虚拟宠物应用，宠物具备以下能力：

1. **可被看见**：具备稳定统一的 2.5D 视觉形象。
2. **可被互动**：支持喂食、抚摸、聊天、玩耍等互动。
3. **可被记住**：支持互动内容与成长过程持久化存储。
4. **可被理解**：宠物根据情绪、属性、上下文产生不同反应。
5. **可被扩展**：后续可接语音、AI、传感器与多端同步能力。

### 2.2 本阶段范围

本次文档聚焦 **宠物表现层由 3D 调整为 2.5D** 带来的功能需求更新，不改变以下核心业务目标：

- 宠物养成
- 情绪系统
- 六维属性系统
- 互动行为系统
- AI 对话与记忆系统
- 数据持久化

---

## 三、2.5D 宠物表现需求

### 3.1 宠物展示形式

宠物主展示区应采用 **2.5D 宠物舞台** 方案：

| 层级 | 内容 | 作用 |
|------|------|------|
| 背景层 | 渐变背景、氛围光、浅装饰图形 | 提供空间氛围 |
| 地台层 | 椭圆阴影、柔光底座 | 提供“站立在场景中”的落点 |
| 主体层 | 宠物主图 / 动作帧 | 作为视觉核心 |
| 前景层 | 对话气泡、爱心、状态提示、粒子特效 | 强化互动反馈 |

### 3.2 基础视觉要求

- 宠物主体必须居中展示，默认占宠物区域的主要视觉焦点。
- 必须具备基础呼吸感或漂浮感，避免静止僵硬。
- 必须提供地影，且地影随缩放轻微变化。
- 必须保证页面为中文文案。
- 图标统一采用极简 SVG 风格，不使用 emoji 图标。
- 容器遵循圆角、阴影、8vp 栅格与统一间距规范。

### 3.3 基础动作表现

2.5D 宠物至少支持以下动作状态：

| 动作 | 说明 | 触发方式 |
|------|------|----------|
| 待机 | 默认循环动作，带轻微呼吸与漂浮 | 页面停留 |
| 关注 | 用户触摸、呼唤后抬头或偏头 | 点击 / 唤起 |
| 开心 | 互动成功后的反馈动作 | 喂食 / 抚摸 / 聊天 |
| 吃东西 | 喂食时播放动作帧或姿态切换 | 喂食 |
| 困倦 | 精力较低时切换为缓动作 | 状态驱动 |
| 睡觉 | 夜间或休息时使用低动态表现 | 时间 / 手动触发 |

### 3.4 交互反馈需求

#### 3.4.1 点击反馈
- 点击宠物后，宠物需要有轻微压缩或抬头反馈。
- 可点击按钮必须具备缩放反馈或颜色反馈。

#### 3.4.2 拖动反馈
- 用户水平拖动宠物时，仅允许小角度偏转。
- 不允许出现完整 360° 自由旋转的 3D 行为。
- 松手后应自动回正。

#### 3.4.3 缩放反馈
- 支持轻量缩放手势或程序缩放反馈。
- 缩放只用于互动表现，不作为核心浏览方式。
- 结束交互后自动恢复默认比例。

#### 3.4.4 对话反馈
- 宠物说话时，头顶展示气泡或附近展示对话容器。
- 对话状态可叠加口型替代动画、轻微跳动、光圈扩散等效果。

## 四、功能模块需求调整

### 4.1 宠物精灵系统

原先偏 3D 资源和 3D 模型驱动的方案，统一调整为 **2.5D 精灵驱动**：

- 资源主形态为：PNG / WebP / SVG / 序列帧。
- 每个动作可采用单图姿态切换或短序列帧方案。
- 支持按宠物品类扩展不同皮肤与主题。
- 不要求必须存在 `glb` 模型资源。

### 4.2 动作状态机

保留“待机、互动、进食、玩耍、睡眠”等状态机思想，但状态输出改为 2.5D 视觉指令：

- 当前动作 ID
- 是否显示气泡
- 当前特效类型
- 当前偏转角度
- 当前缩放倍率
- 是否显示前景装饰

### 4.3 情绪系统映射

情绪系统需要直接映射到 2.5D 表现层：

| 情绪 | 视觉表现 |
|------|----------|
| 开心 | 上浮、轻跳、暖色高光、爱心粒子 |
| 平静 | 缓慢呼吸、低频摆动 |
| 焦虑 | 轻微抖动、汗滴提示、缩小姿态 |
| 悲伤 | 下沉、低饱和、缓慢动作 |
| 困倦 | 半闭眼、下沉、漂浮减弱 |

### 4.4 互动行为系统映射

| 互动类型 | 2.5D 表现 |
|----------|-----------|
| 抚摸 | 主体放大一点、闭眼、爱心或呼噜提示 |
| 喂食 | 食物图标飞入 + 吃东西动作 + 满足气泡 |
| 玩耍 | 小幅跳跃、追逐轨迹或星星特效 |
| 洗澡 | 泡泡特效、清洁后高亮反馈 |
| 聊天 | 对话气泡、说话状态、轻量口型替代动画 |
| 睡觉 | 暗化背景、月光氛围、呼吸节奏变慢 |

## 五、页面需求

### 5.1 首页宠物舞台

首页是 2.5D 宠物的核心展示页，需满足：

- 中部展示宠物舞台区域。
- 顶部可显示当前状态或简短问候。
- 下方保留操作入口，如喂食、玩耍、聊天、档案、设置。
- 宠物舞台区域支持基础手势与互动反馈。
- 在不同状态下，舞台背景和前景特效可切换。

### 5.2 档案页与详情页

档案页中的宠物展示图统一改为 2.5D 静态形象或轻动态卡片：

- 展示宠物头像、名称、品种、领养时间。
- 展示成长阶段与代表性状态。
- 不要求使用 3D 模型预览。

### 5.3 互动页

互动页可在 2.5D 宠物舞台基础上叠加更明确的操作控件：

- 食物卡片
- 玩具卡片
- 聊天输入区
- 快捷互动按钮

## 六、资源需求

### 6.1 资源类型

建议资源优先级如下：

1. 宠物主形象静态图
2. 常用动作短序列帧
3. 情绪特效图层
4. 场景背景与地影资源
5. SVG 图标资源

### 6.2 命名与组织建议

建议资源按宠物类型、动作、情绪分类：

- `pet/cat/idle/`
- `pet/cat/happy/`
- `pet/cat/eat/`
- `pet/common/effects/`
- `icons/actions/`

---

## 七、非功能需求

### 7.1 性能要求

- 首页宠物展示必须在手机端与 Previewer 中稳定运行。
- 不依赖重型 3D 渲染能力作为主链路。
- 高频交互下不得出现明显卡顿或大范围闪烁。
- 动画状态切换应平滑。

### 7.2 可维护性要求

- 宠物表现逻辑与业务逻辑解耦。
- 宠物动作、情绪、特效需有明确枚举或类型定义。
- 禁止使用 `any`。
- 页面、组件、模型、服务应按职责拆分。

### 7.3 可扩展性要求

2.5D 方案后续应支持平滑扩展到：

- AI 对话状态联动
- 语音口播反馈
- 传感器驱动的情绪变化
- 多宠物皮肤扩展
- 多端同步展示

---

## 八、验收标准

### 8.1 视觉验收

- 宠物具备明确的层次感，而不是单一平面静图。
- 宠物舞台具备地影、漂浮、偏转或特效中的至少三种表现。
- 页面整体符合中文化、统一配色、统一圆角阴影规范。

### 8.2 交互验收

- 点击、拖动、缩放至少三类交互反馈可用。
- 喂食、抚摸、聊天至少三类行为可触发不同宠物状态。
- 说话气泡与宠物反馈能同步出现。

### 8.3 工程验收

- ArkTS 严格模式可通过。
- 不依赖废弃布局方式。
- 不以 3D 模型加载成功作为页面可用前提。
- 在现有项目组件结构下可直接演进落地。

## 十、与原 3D 方案的替代关系

| 原方案能力 | 2.5D 替代方式 |
|-----------|---------------|
| 3D 模型展示 | 主图 + 分层舞台 + 小角度偏转 |
| 3D 骨骼动画 | 序列帧 / 姿态切换 / 状态动画 |
| 3D 缩放旋转 | 轻量 scale + rotate + translate |
| 3D 光照层次 | 高光贴图 + 阴影 + 渐变背景 |
| 复杂 3D 设备依赖 | 标准 ArkUI 组件直接实现 |

结论：本次方案调整后，产品核心价值将从“3D 技术展示”转向“稳定、细腻、可持续迭代的陪伴体验”。
