# 虚拟数字宠物 — 技术架构

> 项目名称：**鸿蒙虚拟数字宠物（HarmoPet）**  
> 架构策略：**软件为主，硬件待定** — 模拟器(Previewer)/手机单端完整运行全部核心功能；SC-3568HA全息端 + Hi3863南向端为可选增强（硬件待定，视获取情况决定是否集成）

### 3.2 核心技术选型

| 领域 | 技术方案 | 说明 |
|------|----------|------|
| 开发语言 | ArkTS | 基于OpenHarmony SDK的声明式开发语言，TypeScript超集 |
| UI框架 | ArkUI（声明式） | 组件化开发，@Component装饰器 |
| 运行系统 | OpenHarmony / SwanLinkOS 5.0 | 基于开源鸿蒙系统，符合赛题要求 |
| 状态管理 | @State / @Link / AppStorage | 响应式数据驱动UI更新 |
| 本地存储 | PersistentStorage + 关系型数据库(RDB) | 轻量KV + 结构化数据（**RDB永久存储全部互动记录**） |

| 动画系统 | Animator / Canvas / Lottie | 宠物精灵帧动画 + 粒子特效 |

| 设备通信 | @ohos.bluetooth / Socket | 与南向设备数据交互 |
| 分布式 | @ohos.distributedDeviceManager | 双端设备发现与数据同步 |
| 全息渲染 | Canvas 4方向分屏绘制 | 全息金字塔专用4面画面合成 |

## 七、核心架构风险与降级策略

### 7.1 AI服务依赖

| 场景 | AI在线 | AI离线（降级模式） |
|------|--------|-------------------|
| 宠物动画/互动 | ✅ 正常 | ✅ 正常（纯本地） |
| 属性/情绪系统 | ✅ 正常 | ✅ 正常（纯本地） |
| 互动数据存储 | ✅ RDB+AI双写 | ✅ RDB本地写入 |
| AI对话 | ✅ 完整对话 | ⚠️ 预设回复池 |
| 语音交流 | ✅ 完整 | ❌ 降级为文字 |
| 情绪共鸣(摄像头) | ✅ 可用 | ❌ 降级行为分析 |
| 做梦系统 | ✅ 可用 | ❌ 跳过，恢复后补 |

### 7.2 双端同步风险

| 风险 | 应对 |
|------|------|
| WiFi断连 | 手机端和全息端各自独立运行，重连后增量同步 |
| 动画帧延迟 | 全息端预缓存动画资源，仅同步状态指令而非帧数据 |
| 分布式能力不可用 | 降级为HTTP/WebSocket直连 |

### 7.3 全息金字塔制作风险

| 风险 | 应对 |
|------|------|
| 亚克力加工精度 | 网购成品金字塔（淘宝~30元），不自裁 |
| 显示效果差 | 调整屏幕亮度+对比度，优化4方向画面布局 |
| 比赛运输损坏 | 备用亚克力片，或现场用简易纸板版替代 |
