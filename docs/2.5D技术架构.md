# 虚拟数字宠物 — 2.5D 技术架构文档

> 项目名称：**鸿蒙虚拟数字宠物（HarmoPet）**  
> 架构策略：**以 2.5D 轻量表现替代 3D 主链路**  
> 技术基线：**HarmonyOS NEXT / ArkTS 严格模式 / Stage 模型 / API 12+**

---

## 一、整体架构概览

```text
┌──────────────────────────────────────────────────────────────┐
│                        北向手机应用                           │
│                 ArkTS + ArkUI 声明式界面                     │
│  ┌────────────┐ ┌────────────┐ ┌────────────┐ ┌──────────┐  │
│  │ 2.5D宠物舞台 │ │ 互动操作区   │ │ 聊天与记忆区 │ │ 档案与设置 │  │
│  └────────────┘ └────────────┘ └────────────┘ └──────────┘  │
│  负责：宠物展示、交互反馈、状态呈现、页面导航                  │
└──────────────────────────┬───────────────────────────────────┘
                           │
┌──────────────────────────▼───────────────────────────────────┐
│                         业务服务层                            │
│  PetStateService | PetActionService | EmotionService        │
│  MemoryService   | AIService        | DeviceEventService    │
└──────────────────────────┬───────────────────────────────────┘
                           │
┌──────────────────────────▼───────────────────────────────────┐
│                          数据层                               │
│   @Observed Model / AppStorage / PersistentStorage / RDB     │
└──────────────────────────┬───────────────────────────────────┘
                           │
┌──────────────────────────▼───────────────────────────────────┐
│                     外部扩展与可选能力                         │
│   AI服务后端 / 语音能力 / 南向传感器 / 多端同步                │
└──────────────────────────────────────────────────────────────┘
```

### 1.1 架构目标

本次调整后的架构目标如下：

- 将宠物表现核心收敛到 **ArkUI 标准能力可稳定实现** 的范围。
- 保留情绪、成长、互动、记忆等业务系统不变。
- 让表现层不再依赖重型 3D 模型加载与复杂渲染链路。
- 形成“**业务状态驱动 2.5D 视觉状态**”的清晰分层结构。

### 1.2 设计原则

- **表现轻量**：以图层、阴影、缩放、漂浮、轻旋转实现空间感。
- **业务解耦**：动作状态和属性变化由服务层驱动，不由页面零散拼装。
- **资源友好**：优先使用静态图、序列帧、SVG、特效图层。
- **严格类型**：所有状态模型和动作枚举都使用 ArkTS 严格类型。
- **可降级**：即使缺少高级图形能力，首页宠物舞台依然可正常工作。

## 二、2.5D 表现层架构

### 2.1 组件分层

建议将宠物表现层拆为以下组件：

```text
PetStage
├── StageBackground      背景氛围层
├── PetShadow            地影层
├── PetSprite            宠物主体层
├── PetEffectLayer       前景特效层
└── PetSpeechBubble      对话气泡层
```

### 2.2 组件职责

| 组件 | 职责 |
|------|------|
| `PetStage` | 容器组件，组合各图层并接收页面交互 |
| `PetShadow` | 根据当前缩放与动作调整地影大小和透明度 |
| `PetSprite` | 根据动作状态渲染宠物主体图像或序列帧 |
| `PetEffectLayer` | 渲染爱心、星星、汗滴、气泡等短时特效 |
| `PetSpeechBubble` | 显示宠物当前说话内容或提示信息 |

### 2.3 渲染策略

宠物舞台不采用完整 3D 场景树，而采用 **Stack 分层渲染**：

- 背景使用渐变色、柔光斑或轻纹理；
- 地影使用 `Ellipse` 或图片阴影资源；
- 宠物主体使用 `Image` 或序列帧组件；
- 前景特效使用 `ForEach + Image/Text` 进行短时动画绘制；
- 对话内容使用普通 ArkUI 容器组件呈现。

这使得宠物表现与普通页面布局处于同一 UI 体系内，便于维护和调试。

## 三、状态驱动架构

### 3.1 核心思路

业务层不直接控制每一个视觉 API，而是统一输出 **宠物展示状态**。

```text
用户操作 / 定时器 / AI回复 / 传感器事件
                ↓
          业务服务层计算
                ↓
       生成 PetPresentationState
                ↓
        2.5D 组件根据状态渲染
```

### 3.2 推荐状态模型

```ts
export type PetActionType = 'idle' | 'focus' | 'happy' | 'eat' | 'sleep' | 'sad' | 'anxious';

export type PetEffectType = 'heart' | 'star' | 'bubble' | 'sweat' | 'none';

export interface PetPresentationState {
  action: PetActionType;
  speechText: string;
  scale: number;
  offsetY: number;
  tiltDeg: number;
  shadowScale: number;
  effect: PetEffectType;
  showBubble: boolean;
}
```

说明：

- `action` 表示当前动作；
- `speechText` 表示当前气泡文案；
- `scale`、`offsetY`、`tiltDeg` 控制 2.5D 视觉反馈；
- `shadowScale` 控制地影变化；
- `effect` 控制前景特效；
- `showBubble` 控制气泡显隐。

### 3.3 与现有模型的衔接

当前项目已存在 [PetModel.ets](entry/src/main/ets/common/models/PetModel.ets) 中的 `currentAction`、`currentSpeech`、`image`、`modelPath` 等字段。

建议演进方向：

- 保留 `currentAction` 与 `currentSpeech`；
- 弱化 `modelPath` 在主链路中的作用；
- 新增或映射出专门的 `PetPresentationState`；
- 页面层只消费展示状态，不直接操作底层资源细节。

## 四、页面层架构

### 4.1 首页结构建议

首页建议拆分为：

```text
Index / MainPetPage
├── 顶部状态栏
├── 2.5D宠物舞台区
├── 属性概览区
├── 快捷互动菜单
└── 扩展信息区
```

其中，2.5D 宠物舞台区是页面核心展示区，推荐封装成独立组件，以便在首页、聊天页、档案页复用。

### 4.2 档案页结构建议

档案页中的宠物展示建议复用静态化舞台卡片，而不是单独再维护一套高成本展示链路：

- 头像或 2.5D 静态立绘；
- 成长标签；
- 当前状态；
- 基础属性信息。

### 4.3 操作菜单结构建议

当前 [PetActionMenu.ets](entry/src/main/ets/common/components/PetActionMenu.ets) 已具备底部动作入口，可继续沿用其结构，但建议后续增强：

- 加入 SVG 图标资源；
- 与当前动作状态联动高亮；
- 触发 `PetActionService`，而不是直接在页面里散落逻辑。

## 五、组件设计与现有代码映射

### 5.1 现有组件评估

当前 [PetSprite.ets](entry/src/main/ets/common/components/PetSprite.ets) 已经具备以下基础：

- 3D 与 2D 的分支判断；
- 语音气泡展示；
- 阴影层；
- 拖动与捏合手势；
- 模拟器降级逻辑。

这为迁移到 2.5D 提供了直接基础。

### 5.2 建议改造方向

将当前 `PetSprite` 从“3D 优先 + 2D 降级”调整为“**2.5D 主链路 + 3D 可选试验能力**”：

- 默认直接渲染 2.5D 主体图层；
- `SceneView` 不再作为首页主路径依赖；
- 手势逻辑保留，但只输出轻量偏转与缩放；
- 将阴影、主体、气泡、特效拆分为更清晰的小组件。

### 5.3 推荐组件接口

```ts
@Component
export struct PetStage {
  @ObjectLink pet: PetModel;
  @Prop presentation: PetPresentationState;
}
```

或者：

```ts
@Component
export struct PetSprite {
  @ObjectLink pet: PetModel;
  @Prop action: PetActionType;
  @Prop speechText: string;
  @Prop effect: PetEffectType;
}
```

重点是：**组件接收清晰状态，不自行推导复杂业务规则**。

## 六、动作与动画架构

### 6.1 动画层次

建议把动画拆为三层：

1. **常驻动画**：呼吸、漂浮、轻微摆动。
2. **动作动画**：开心、吃东西、睡觉、抚摸反馈。
3. **特效动画**：爱心、星星、汗滴、说话气泡。

### 6.2 动作调度原则

动作优先级建议：

```text
用户即时互动 > AI回复反馈 > 属性触发状态 > 待机循环
```

例如：

- 用户喂食时进入 `eat`；
- 动作结束后回到 `happy` 或 `idle`；
- 若精力过低，则优先进入 `sleep` 或 `tired`；
- 若 AI 刚生成回复，则短时显示 `focus` + `showBubble=true`。

### 6.3 资源组织建议

```text
resources/
└── base/
    └── media/
        ├── pet/
        │   ├── cat/
        │   │   ├── idle.png
        │   │   ├── happy.png
        │   │   ├── eat_01.png
        │   │   ├── eat_02.png
        │   │   └── sleep.png
        │   └── common/
        ├── effects/
        │   ├── heart.svg
        │   ├── star.svg
        │   └── sweat.svg
        └── icons/
            └── actions/
```

## 七、服务层架构

### 7.1 服务拆分建议

| 服务 | 职责 |
|------|------|
| `PetStateService` | 管理宠物属性、阶段、当前状态 |
| `PetActionService` | 响应喂食/抚摸/聊天等操作，生成动作切换 |
| `EmotionService` | 根据属性、环境、上下文推导情绪 |
| `MemoryService` | 持久化互动日志、对话、成长数据 |
| `AIService` | 对接 AI 能力，输出对话与情绪建议 |
| `DeviceEventService` | 对接传感器与设备事件 |

### 7.2 `PetActionService` 职责重点

2.5D 方案下，最关键的是把“动作触发”从页面剥离出来：

- 接收用户动作：喂食、抚摸、玩耍、聊天；
- 计算当前展示状态：动作、气泡、特效；
- 控制短时状态切换与回退；
- 通知页面进行刷新。

这样可以避免各页面重复写动画切换逻辑。

## 八、数据层架构

### 8.1 数据存储策略

继续沿用当前文档中的双层持久化思想：

- `PersistentStorage`：存放宠物基础信息、当前配置、轻量状态；
- `RDB`：存放互动日志、聊天记录、情绪记录、成长数据；
- AI 后端：作为增强能力，不作为唯一数据源。

### 8.2 数据类型建议

所有关键结构都应使用明确类型：

```ts
export interface InteractionRecord {
  id: string;
  type: 'feed' | 'pet' | 'play' | 'chat' | 'sleep';
  content: string;
  createdAt: number;
}
```

```ts
export interface EmotionSnapshot {
  emotion: 'happy' | 'calm' | 'sad' | 'anxious' | 'sleepy';
  moodValue: number;
  energyValue: number;
  timestamp: number;
}
```

## 九、性能与降级策略

### 9.1 性能优化原则

- 以图片与基础动画为主，避免首页加载重型场景资源。
- 高频动作资源优先预加载。
- 统一动画时长和曲线配置，减少分散开销。
- 对短时特效数量进行限制，避免同时创建过多节点。

### 9.2 降级策略

| 场景 | 处理方式 |
|------|----------|
| 资源未加载完成 | 显示静态占位图或基础待机图 |
| 特效资源缺失 | 仅展示宠物主体与气泡 |
| 低性能设备 | 关闭部分前景粒子与复杂渐变 |
| AI 服务不可用 | 使用本地固定回复与基础状态逻辑 |

2.5D 方案本身就是对“3D 主链路”的工程级降复杂度优化。

## 十、实施路径建议

### Phase 1：建立 2.5D 主链路
- 重构 `PetSprite` 为 2.5D 主实现
- 抽出地影、气泡、特效层
- 用现有 `PetModel` 驱动基础动作

### Phase 2：统一动作与情绪映射
- 建立 `PetPresentationState`
- 建立 `PetActionService`
- 打通喂食 / 抚摸 / 聊天的状态联动

### Phase 3：补充资源与页面复用
- 补充动作帧和 SVG 图标
- 首页、档案页、互动页统一宠物展示逻辑
- 优化主题与视觉一致性

### Phase 4：接入扩展能力
- AI 对话联动
- 语音状态联动
- 传感器事件驱动
- 多端同步展示

---

## 十一、结论

本次技术架构调整的核心，不是单纯把 3D 去掉，而是将宠物表现层重构为：

- **更稳**：更适合 HarmonyOS NEXT 当前项目节奏；
- **更轻**：更利于 Previewer、真机与后续维护；
- **更清晰**：表现层、业务层、数据层职责更明确；
- **更可扩展**：后续仍可接 AI、语音、传感器和多端能力。

对于当前仓库而言，建议以 [PetSprite.ets](entry/src/main/ets/common/components/PetSprite.ets)、[PetModel.ets](entry/src/main/ets/common/models/PetModel.ets) 和 [PetActionMenu.ets](entry/src/main/ets/common/components/PetActionMenu.ets) 作为 2.5D 化改造的起点。
